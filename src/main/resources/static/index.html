Week 1

Arquitectura y Gestión en Sistemas Operativos

Procesos, Memoria y Algoritmos de Planificación

🎛️

El Orquestador: Definición y Propósito

El Sistema Operativo (SO) es la capa de software que administra el hardware y proporciona una interfaz abstracta para las aplicaciones. Actúa como intermediario entre el 'lenguaje de máquina' y el usuario.

  • Multiplexación en el Tiempo: Turnos rotativos de uso (ej. CPU).
  • Multiplexación en el Espacio: División física de recursos (ej. RAM).
  • Clasificación: Batch, Tiempo Compartido, Tiempo Real.
  • Capas: Aplicaciones → SO (Gestión) → Hardware (Recursos Físicos).
🖥️

Abstracción del Hardware: La Máquina Virtual

El VMM (Monitor de Máquina Virtual) crea copias exactas del hardware, permitiendo aislamiento y consolidación entre sistemas.

  • Tipo 1 — Bare Metal: Hipervisor directo sobre hardware. Ej: VMware ESXi.
  • Tipo 2 — Hosted: Hipervisor sobre un SO anfitrión. Ej: VirtualBox.
  • Aislamiento: Cada VM opera de forma independiente.
  • Consolidación: Múltiples sistemas en un solo servidor físico.
⚙️

Anatomía de la Unidad de Trabajo: El Proceso

Un programa es una entidad pasiva (código en disco). Un proceso es una entidad activa (programa en ejecución). Cada proceso tiene un PCB con toda su información de estado.

  • PID: Identificador único del proceso (ej: 0x812A).
  • Estado: Listo / En Ejecución / Bloqueado.
  • Contador de Programa & Registros CPU: Indican la siguiente instrucción.
  • Límites de Memoria: Rango de direcciones asignadas (ej: 0x8000 – 0xFFFF).
  • Info de Contabilidad: Uso de CPU, estadísticas de I/O.
🔄

Ciclo de Vida: Diagrama de Estados del Proceso

Los procesos transitan entre estados. Solo puede haber 1 proceso en ejecución por CPU al mismo tiempo.

  • Nuevo → Ready: El proceso es admitido al sistema.
  • Ready → Run: El dispatcher asigna la CPU (Dispatch).
  • Run → Wait: El proceso solicita una operación E/S.
  • Wait → Ready: Fin de la operación E/S.
  • Run → Fin: El proceso termina su ejecución (Exit).
  • Run → Ready: Fin de tiempo / Interrupción expulsiva.
🍴

Control del Ciclo de Vida: UNIX vs. Windows

Los SO exponen llamadas al sistema para crear, ejecutar y terminar procesos. UNIX y Windows difieren en su modelo de creación.

  • Creación UNIX: fork() — copia exacta del padre.
  • Creación Windows: CreateProcess() — espacio completamente nuevo.
  • Ejecución nueva UNIX: execve() — reemplaza la imagen del proceso.
  • Terminación: exit/kill (UNIX) · ExitProcess/TerminateProcess (Windows).
  • Sincronización: waitpid() (UNIX) · WaitForSingleObject() (Windows).
📅

El Estratega: Introducción a la Planificación de CPU

El objetivo del planificador es maximizar el uso de CPU, la equidad y el throughput. Existen tres niveles según el horizonte de tiempo.

  • Largo Plazo (Admisión): Decide qué procesos ingresan (~300 ms).
  • Medio Plazo (Swapping): Mueve procesos entre RAM y disco.
  • Corto Plazo (Dispatcher): Asigna la CPU al proceso listo (~50 ms).
  • Expulsiva: El SO puede interrumpir al proceso activo.
  • No Expulsiva: El proceso cede la CPU voluntariamente.
📊

Algoritmos Básicos: FCFS y SJF

Los algoritmos de planificación determinan el orden de ejecución. FCFS es simple pero genera el efecto convoy. SJF minimiza el tiempo de espera pero puede causar inanición.

  • FCFS: No expulsivo. Tiempo de espera prom.: 2.75. Sufre efecto convoy.
  • SJF: No expulsivo. Tiempo de espera prom.: 1.5. Óptimo pero requiere duración a priori.
  • Efecto Convoy: Procesos cortos esperan detrás de uno largo.
  • Inanición: Procesos largos pueden esperar indefinidamente en SJF.
🔁

Algoritmos Avanzados: SRTF y Round Robin

SRTF es la versión expulsiva de SJF. Round Robin garantiza equidad con un quantum fijo, ideal para sistemas de tiempo compartido.

  • SRTF: Expulsivo. Si llega un proceso más corto, interrumpe al actual.
  • Round Robin: Cada proceso recibe un quantum fijo (Quantum = 4).
  • RR — Espera prom.: 0.5 (excelente respuesta interactiva).
  • RR — Retorno prom.: 4.25 (mayor latencia por cambios de contexto).
  • Elección del Quantum: Muy pequeño → muchos cambios. Muy grande → como FCFS.
💾

El Espacio de Trabajo: Gestión de Memoria

La gestión de memoria busca proteger los procesos entre sí, permitir la compartición controlada y optimizar el uso para maximizar la multiprogramación.

  • Protección: Aislamiento entre procesos — ninguno accede a memoria ajena.
  • Compartición: Acceso común controlado a zonas compartidas (librerías, IPC).
  • Optimización: Maximizar la multiprogramación con más procesos en RAM.
  • Jerarquía: Registros/Caché → RAM (volátil) → Disco (persistente).
🗺️

La Gran Ilusión: Memoria Virtual y Paginación

La MMU traduce direcciones lógicas a físicas. Las páginas no usadas se mueven al área de Swap en disco, creando la ilusión de memoria infinita.

  • MMU + TLB: Traduce direcciones virtuales a físicas. TLB es caché de traducciones recientes.
  • Paginación: Memoria dividida en páginas (lógicas) y marcos (físicos) de tamaño fijo.
  • Fallo de Página: La página no está en RAM → se carga desde el disco (Swap).
  • Memoria Virtual: Cada proceso tiene su propio espacio lógico, mayor que la RAM física.

Definición de Hilos

Un hilo es una línea de ejecución dentro de un proceso. Todo proceso inicia con un único hilo principal, pero el sistema operativo permite crear múltiples hilos que comparten el código y los datos, aunque cada uno mantiene su propio *Stack* de ejecución.

Resumen: El Sistema Equilibrado

Week 2

Evolución de los Sistemas Operativos

De los Laboratorios Bell a la Nube: Historia y ecosistema de UNIX, Linux, macOS y Windows.

🧠

El Intermediario Esencial

Los sistemas operativos son el componente fundamental que permite la interacción entre los usuarios, el hardware y el software[cite: 321].

  • Impulsores de Evolución: Avances tecnológicos[cite: 323].
  • Cambios en las necesidades de procesamiento[cite: 324, 325].
  • La consolidación de nuevos paradigmas de diseño[cite: 326, 327].
🏛️

El Génesis: UNIX (1969)

Desarrollado en los laboratorios Bell de AT&T por Ken Thompson, Dennis Ritchie y colaboradores[cite: 330].

  • Arquitectura Modular: Diseño multiusuario y multitarea[cite: 332].
  • Portabilidad: Su reescritura en lenguaje C permitió llevarlo a diferentes plataformas[cite: 333].
  • Legado: Derivó en sistemas como BSD, System V, Solaris, HP-UX y AIX[cite: 337, 338, 340, 341, 342].
🐧

La Revolución Open Source: Linux

Iniciado en 1991 por Linus Torvalds, liberado bajo licencia GPL[cite: 346].

  • Dominio: Lidera en computación en la nube, servidores y supercomputadores[cite: 351].
  • Móviles: Es la base de Android, el SO móvil más usado[cite: 352].
  • Distribuciones: Ubuntu, Debian, Fedora, Red Hat Enterprise[cite: 360].
🍎

Apple: Elegancia sobre UNIX

Mac OS X (2001) integra una interfaz gráfica avanzada con la potencia de un sistema UNIX[cite: 363].

  • Fundamentos: Basado en Darwin[cite: 364].
  • Componentes: Utiliza elementos de BSD y el microkernel Mach[cite: 364].
  • Convergencia: Logra una experiencia altamente optimizada y modular[cite: 365].
🪟

La Ruta Paralela: Microsoft

Comenzó en 1981 con MS-DOS (línea de comandos) y evolucionó hacia interfaces gráficas[cite: 370, 371].

  • Estandarización: Windows 95 introdujo el menú Inicio y Plug and Play[cite: 390, 391, 393].
  • Estabilidad: Windows NT/2000 aportó arquitectura empresarial[cite: 398, 399].
  • Convergencia: Windows 11 integra compatibilidad con Android y Linux mediante WSL 2[cite: 415].
☁️

El Futuro de los Sistemas Operativos

La innovación tecnológica ha forzado la transición desde las arquitecturas monolíticas del pasado hacia sistemas modernos más flexibles[cite: 432].

  • Nuevas Arquitecturas: Sistemas portables, modulares y distribuidos[cite: 432].
  • Tendencias: Virtualización y computación en la nube[cite: 434, 435].
  • Retos: Seguridad continua y convergencia de tecnologías[cite: 433, 436].
Week 3

Historia, Ciclos y Decisiones del Sistema

Evolución de las arquitecturas de SO y análisis práctico mediante simulación de algoritmos de planificación de CPU.

🖥️

Evolución: De Monolíticos a Multiproceso

La informática ha evolucionado drásticamente desde los años 40[cite: 466].

  • Monolíticos (50s-70s): Código en un solo bloque ejecutando un proceso a la vez en mainframes[cite: 469, 470].
  • Multitarea: Popularizada por UNIX, alterna tareas tan rápido que parecen simultáneas[cite: 473].
  • Multiprocesamiento: Distribuye tareas entre múltiples núcleos de CPU para agilizar procesos[cite: 476].
☁️

Distribución, Virtualización y Nube

Las arquitecturas modernas rompieron las barreras del hardware físico.

  • Distribuidos: Conectan múltiples computadoras (nodos) para compartir recursos y procesar datos masivos[cite: 481].
  • Virtualización: Usa hipervisores para ejecutar múltiples SO en un solo hardware de forma aislada[cite: 483].
  • Cloud Computing: Gestiona recursos de forma elástica en centros de datos globales[cite: 486].
⏱️

Planificación: FIFO y SJF

Estrategias fundamentales para la asignación de la CPU a los procesos.

  • First In, First Out (FIFO): El primer proceso que llega es el primero en ejecutarse[cite: 491]. Genera el "efecto convoy"[cite: 491].
  • Shortest Job First (SJF): Asigna el CPU al proceso más corto, reduciendo el tiempo promedio de espera[cite: 496].
🔄

Planificación: Round Robin y Colas

Algoritmos diseñados para la equidad y la priorización.

  • Round Robin: Algoritmo preventivo que asigna un quantum fijo a cada proceso y los ejecuta por turnos[cite: 493].
  • Multilevel Queue: Divide procesos en colas por nivel de prioridad, mejorando el control de tareas críticas[cite: 499].

🧠

Reflexión y Conclusión

El diseño de sistemas operativos busca un equilibrio constante entre el rendimiento técnico, la justicia en la asignación y la adaptabilidad[cite: 550].

  • No existe una solución única; su efectividad es contextual[cite: 548].
  • SJF: Ideal en entornos donde el objetivo principal es minimizar el tiempo de promedio[cite: 540].
  • Round Robin: Es más justo y ofrece mejor tiempo de respuesta inicial[cite: 542].
